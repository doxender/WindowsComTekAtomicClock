ComTek Atomic Clock — Windows
Requirements specification
==========================================

Last updated: 2026-04-25

Note: this file is a project requirements document, not a Python pip
manifest. .NET dependencies are declared in each project's .csproj file.
This document is the source of truth for what the product must do; it
is updated whenever a scope, architecture, or tooling decision changes.

------------------------------------------------------------------
1. FUNCTIONAL REQUIREMENTS
------------------------------------------------------------------

1.1  Clock display
     - The product ships with eleven face themes (subject to revision)
       organized into three categories. Visual references and design
       rationale for each live under design/themes/ in this repo.

       (a) Analog with integrated digital readout:
           - Atomic Lab    (default on first run)
           - Boulder Slate
           - Aero Glass
           - Cathode
           - Concourse
           - Daylight

       (b) Digital-only:
           - Flip Clock      (retro mechanical, animated card flip)
           - Marquee         (theater bulbs, animated chase)
           - Slab            (brutalist concrete, static)
           - Binary Digital  (binary HH/MM/SS as text, terminal
                              aesthetic; pairs with the BCD encoding
                              face below for users who want the
                              binary read-out without the dot grid)

       (c) Specialty encodings:
           - Binary      (BCD LED stack with decoded readout)
           - Hex         (hex per unit + 16-bit day fraction shown
                          as a literal #RRGGBB color swatch)

     - The active theme is selectable per clock window and per tab
       (see § 1.2 for the per-tab settings gear and § 1.8 for the
       global "same theme everywhere" override).
     - Per-theme color customization. Every theme — analog,
       digital-only, AND encoding — exposes the same five
       user-overridable color slots, set per tab (or per
       free-floating window) via the settings gear popover (§ 1.2):

         · Ring      - outermost frame / bezel / case
         · Face      - dial background / panel / tile face
         · Hands     - hour + minute + second hands (analog) or
                       primary accent strokes (digital, encoding)
         · Numbers   - numerals (analog) or primary big digits
                       (digital, encoding)
         · Digital   - integrated digital readout color (analog) or
                       secondary/auxiliary digital text such as
                       seconds, decoded readout, footer captions
                       (digital, encoding)

       Each theme's renderer maps the five slots to the closest
       visual element for that theme; every theme implements all
       five slots so the settings UI is identical across theme
       types. design/README.md documents the per-theme
       slot-to-element mapping. The settings popover shall offer a
       "Reset colors to theme defaults" action.

       Theme-specific accent colors that do not map to any of the
       five slots (e.g., Boulder Slate's red second-hand disc,
       Atomic Lab's `NIST · BOULDER · CO` legend, Flip Clock's
       amber colon pips) remain at the theme's discretion and are
       NOT user-overridable. These accents are part of each theme's
       identity.
     - Faces shall update at least once per second; analog second
       hand motion may be smooth (sub-second) or stepped at the
       user's choice.
     - For analog themes (category a), the integrated digital readout
       shall be independently hideable per clock window or per tab,
       leaving a pure analog face. The toggle is hidden for themes
       in categories (b) and (c) since they have no analog face to
       reveal.

1.2  Multiple time zones (tabbed view)
     - The main application window shall present a tabbed view, each
       tab bound to a single IANA / Windows time zone.
     - Each tab shall expose a settings gear icon on the right side of
       the tab header. Clicking it opens a popover containing:
         - Time-zone selector (dropdown of IANA / Windows zones).
         - Theme selector (dropdown of all themes from § 1.1). This
           selector is disabled when the global "Use the same theme
           across all tabs" toggle is ON (see § 1.8).
         - "Show digital readout" toggle. Visible only when the
           active theme is in category (a). Hidden for digital-only
           and specialty themes since they have no separable digital
           readout to toggle.
         - Five color pickers labeled "Ring", "Face", "Hands",
           "Numbers", and "Digital" (per § 1.1). Each picker
           overrides the active theme's color for that slot on this
           tab only — unless the global "Use the same theme across
           all tabs" toggle is ON, in which case the override
           propagates to all tabs. A "Reset colors to theme
           defaults" link sits under the pickers.
     - The user can add, rename, reorder, and remove tabs.

1.3  Multiple simultaneous clock windows
     - The user can spawn any number of additional clock windows.
     - Each window is independently configurable for time zone, face
       style, size, and on-top behavior.
     - Window state (position, size, time zone, face) shall persist
       across application restarts.

1.4  Desktop overlay mode
     - A clock window can be placed in "desktop overlay" mode: borderless,
       transparent background, optionally click-through, and pinned to
       the desktop layer (behind normal windows, in front of wallpaper).
     - Multiple overlay clocks may run simultaneously.

1.5  Atomic clock synchronization
     - The application shall synchronize the Windows system clock to a
       NIST time server (default: time.nist.gov, Boulder anycast).
     - Synchronization uses SNTP (RFC 4330) over UDP/123.
     - Default sync interval: hourly. User-configurable between 15
       minutes and 24 hours.
     - Manual "Sync now" action shall be available from the tray menu.
     - The application shall respect the NIST 4-second minimum poll
       interval per server.

1.6  Background service
     - System clock synchronization shall run as a Windows Service so
       that it continues without an interactive user session.
     - Service start mode: Automatic.
     - Service shall recover automatically from transient network or
       DNS failures (exponential backoff, capped retry).

1.7  System tray integration
     - The UI shall provide a tray icon with: open main window, sync
       now, last-sync status, settings, exit.
     - Closing the last visible window shall minimize to tray, not
       terminate the UI process.

1.8  Settings
     - Per-tab: time zone, theme, "show digital readout" (analog
       themes only), and the five color overrides (ring, face,
       hands, numbers, digital).
     - Per-window (clock windows spawned outside the main tabbed
       view per § 1.3): time zone, theme, size, position, on-top,
       overlay mode, "show digital readout" (analog themes only),
       and the five color overrides.
     - Global:
         - Sync server, sync interval.
         - Start with Windows.
         - Service install state.
         - "Use the same theme across all tabs" toggle. When ON,
           changing the theme — or any of its five color overrides
           — on any tab applies to all tabs, and disables the
           per-tab theme and color selectors in their settings
           popovers. When OFF (the default), each tab keeps its own
           theme and its own color overrides. The toggle does NOT
           govern free-floating clock windows from § 1.3 — those
           are always per-window.
         - Default theme on first run: Atomic Lab.

2. NON-FUNCTIONAL REQUIREMENTS
------------------------------------------------------------------

2.1  Platform
     - Windows 10 version 1809 or newer, and Windows 11.
     - x64 and ARM64 builds.

2.2  Runtime and language
     - C# 12 on .NET 8 (LTS).
     - WPF for the UI; Worker Service template for the system service;
       a class library for shared IPC contracts and models.

2.3  Privileges
     - The UI process shall run as the standard user.
     - Only the service requires LocalSystem (to call SetSystemTime).
     - The UI shall NOT request elevation for normal operation.

2.4  Inter-process communication
     - UI <-> Service via named pipe with a versioned message contract.
     - Authentication: pipe ACL restricts access to interactive users
       on the local machine.

2.5  Reliability
     - Service shall log sync attempts, offsets applied, and failures
       to the Windows Event Log under a custom source.
     - Sync offset greater than a configurable threshold (default 5 s)
       is logged at Warning; the user may opt to require confirmation
       before applying large corrections.

2.6  Security
     - All NIST traffic is plain SNTP; the service shall validate the
       NTP packet's mode, version, and stratum fields and reject
       malformed responses.
     - Release binaries shall be Authenticode-signed before
       distribution.

2.7  Packaging
     - Single MSIX installer that deploys both the UI and the service
       and registers the service.
     - Uninstaller shall stop and remove the service.

2.8  Build environment
     - Hard prerequisite: .NET 8 SDK on PATH. The full solution shall
       build, test, and run from the command line via `dotnet build`
       / `dotnet run` with no IDE present.
     - Recommended for human inspection / debugging: Visual Studio
       2026 (or 2022) Community edition with the ".NET desktop
       development" workload. The IDE provides the XAML designer,
       multi-process debugging, and the Windows SDK / `signtool.exe`
       used during release packaging. The IDE is NOT required for
       contributors who only edit code.

3. OUT OF SCOPE (initial release)
------------------------------------------------------------------

     - Cross-platform builds (macOS, Linux).
     - Stratum-1 hardware reference clocks.
     - Authenticated NTP (NTS / symmetric keys).
     - Cloud sync of user settings across machines.
